Why should a software engineer try this?

Because the hard problem is often not "can an LLM write code?" It is "which code should it touch, what should it learn from, what is risky, and how do I preserve the reasoning for the next run?"

01

It turns repo sprawl into an engineering map

Point CAM_CAM at a directory of repos. It inventories them, clusters them by capability, detects duplicates by normalized names, and produces a graph that a human or agent can navigate.

02

It blocks the usual AI-agent footgun

The first action is read-only triage. It reports dirty worktrees, missing tests, security-sensitive language, medical/PII indicators, and API-key-sensitive language before any merge or rewrite.

03

It gives you reusable context, not just a PDF

The output is structured for follow-on work: JSON inventory, graph nodes/edges, GraphRAG chunks, Logseq pages, Markmap outline, Freeplane map, dashboard HTML, risks, and ranked opportunities.

04

It helps choose the next useful build

Instead of forcing an arbitrary demo app, it ranks opportunities from the repo universe. In the real scan, the top recommendation was a Local Knowledge Appliance, followed by Repo Rescue Desk itself.

05

It is a bridge into CAM's learning loop

CAM-PULSE proves mining, retrieval, verification, and learning from outcomes. CAM_CAM's role is to prepare high-quality cross-repo context so that loop can be applied safely to a local fleet.

06

It is useful even before automation

A team can use the generated risk report, opportunities, and mindmaps as a planning artifact before enabling code modification, model calls, or repository assimilation.

How it works technically

Current implementation is deterministic Python. No LLM calls are required for the first triage pass, which keeps it cheap, reproducible, and safe to run on sensitive local repos.

1. Discover

Walk one root directory and select direct child git repos. Skip dependency and build folders such as node_modules, .venv, .next, target, and caches.

2. Profile

Read README text, count language extensions, inspect git remote, branch, short HEAD, last commit date, dirty state, and untracked file count.

3. Cluster

Score repo name plus README against domain keyword families. Assign the strongest cluster or mark as uncategorized when no signal is present.

4. Risk flag

Detect dirty worktree, no README, no tests, medical/PII terms, security terms, API-key terms, and large Node dependency risk.

5. Graph

Create cluster nodes, repo nodes, belongs_to edges, and duplicate-name group edges. The real run emitted 249 nodes and 244 edges.

6. Rank

Score showpiece opportunities by support across required clusters, then attach supporting repo evidence and measurable outcome targets.

7. Export

Write dashboard HTML, JSON inventory, graph JSON, GraphRAG Markdown, Markmap Markdown, Freeplane XML, Logseq pages, risk report, and opportunities.

8. Preserve

Write scan receipts so future CAM steps can prove what source state was inspected before any action is taken.

How this differs from coding assistants

CAM_CAM is not trying to beat an autocomplete tool at typing. It is trying to decide what context and safeguards should exist before an AI engineer acts.
