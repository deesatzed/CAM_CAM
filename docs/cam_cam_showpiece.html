Repo Rescue Desk: current CAM_CAM flagship

Preflight your repo universe before AI agents touch it.

CAM_CAM scans a folder of real repos, finds reusable systems, flags risk before an agent edits code, ranks what should be built or mined next, runs git-safe preflight checks, finds repo-to-repo reuse candidates, answers deterministic follow-up questions, and exports machine-readable context for GraphRAG, Logseq, Markmap, and Freeplane.

A SWE would try this when they have a messy local repo collection, inherited code, experimental agent tools, or a proposed multi-repo merger and need an evidence-backed plan before letting an AI mutate anything.

Why should a software engineer try this?

Because the hard problem is often not "can an LLM write code?" It is "which code should it touch, what should it learn from, what is risky, and how do I preserve the reasoning for the next run?"

01

Preflight for multi-repo AI work

Before an agent edits anything, CAM_CAM inventories the repos, records git state, flags dirty worktrees, missing tests, security-sensitive terms, medical/PII signals, and key/API language.

02

Turn local code sprawl into a map

It clusters repos by capability, finds duplicate-name families, emits a graph, and ranks build opportunities using the systems already sitting in the workspace.

03

Export context CAM-PULSE can use

The output is not a slide deck. It is JSON inventory, graph JSON, GraphRAG chunks, Logseq pages, Markmap Markdown, Freeplane XML, risk reports, opportunities, and scan receipts.

How it works technically

Current implementation is deterministic Python. No LLM calls are required for the first triage pass, which keeps it cheap, reproducible, and safe to run on sensitive local repos.

1. Discover

Walk one root directory and select direct child git repos. Skip dependency and build folders such as node_modules, .venv, .next, target, and caches.

2. Profile

Read README text, count language extensions, inspect git remote, branch, short HEAD, last commit date, dirty state, and untracked file count.

3. Cluster

Score repo name plus README against domain keyword families. Assign the strongest cluster or mark as uncategorized when no signal is present.

4. Risk flag

Detect dirty worktree, no README, no tests, medical/PII terms, security terms, API-key terms, and large Node dependency risk.

5. Graph

Create cluster nodes, repo nodes, belongs_to edges, and duplicate-name group edges. The real run emitted 249 nodes and 244 edges.

6. Rank

Score showpiece opportunities by support across required clusters, then attach supporting repo evidence and measurable outcome targets.

7. Export

Write dashboard HTML, JSON inventory, graph JSON, GraphRAG Markdown, Markmap Markdown, Freeplane XML, Logseq pages, risk report, and opportunities.

8. Preserve

Write scan receipts so future CAM steps can prove what source state was inspected before any action is taken.

How this differs from coding assistants

Copilot, Devin, Windsurf, Cursor-class tools, and Aider all have useful memory, indexing, rules, wiki, or repo-map features. CAM_CAM is narrower: it decides what local repo evidence, mined methods, and safeguards should exist before an AI engineer acts.

Run it locally

Use this when you want concrete answers to: what do I have, what is risky, what can I reuse, what should I do next, and is this repo safe for an agent to edit?

git clone https://github.com/deesatzed/CAM_CAM.git
cd CAM_CAM
PYTHONPATH=apps/repo_rescue_desk python -m repo_rescue_desk.cli \
  --root /path/to/folder/containing/git/repos \
  --out-dir tmp/repo_rescue_desk/latest

open tmp/repo_rescue_desk/latest/repo_rescue_dashboard.html
cat tmp/repo_rescue_desk/latest/executive_report.md

PYTHONPATH=apps/repo_rescue_desk python -m repo_rescue_desk.cli ask \
  --report tmp/repo_rescue_desk/latest/repo_inventory.json \
  --question "What should I do next?"

PYTHONPATH=apps/repo_rescue_desk python -m repo_rescue_desk.cli preflight \
  --repo /path/to/repo \
  --json-out tmp/repo_rescue_desk/preflight.json

Replace --root with any directory whose direct children are git repos, such as ~/Projects, ~/work, or a team checkout folder. The machine-specific proof path is only kept in the reproducibility notes; source repos are not modified.
